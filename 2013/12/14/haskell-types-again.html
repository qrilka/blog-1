Haskell: и снова о типах

(2013, 12, 14)
Haskell

В языке Haskell есть три ключевых слова, используемые при определении собственных типов. И слова эти могут, скажем так, смутить программистов, пришедших из мира C++. Вот эти слова:

  • data
  • class
  • instance

Ключевое слово data служит для определения пользовательского типа (естественно, оно служит также и для определения всех стандартных типов Haskell, но сейчас это не столь важно).

Что же тогда означает слово class, спросите вы. Это слово предназначено для определения класса типов. Да, я понимаю, звучит на первый взгляд странно, но идея этой концепции весьма проста.

В языке C++ понятия “тип” и “класс” фактически взаимозаменяемы, потому что ключевое слово class как раз и используется для определения пользовательских типов. Если мы написали:

class IP_address {
    // ...
};

int main() {
    IP_address address;
}

что это означает? Мы определили класс IP_address, но при этом мы определили и пользовательский тип IP_address. :) Однако в языке Haskell всё иначе. А чтобы моё объяснение было более понятным, сразу перейдём к практическому примеру.

Собственный тип

Для начала определим собственный тип IP_address:

data IP_address = IP_address String

Готово. Перед нами простейший пользовательский тип. Объект этого типа фактически будет представлять собою значение стандартного типа String с меткой IP_address.

Вы спросите, что такое метка? Рассматривайте метку как пояснительный идентификатор. Многие типы (в частности, стандартные типы) не имеют метки, поэтому их можно инициализировать значениями без пояснений. Например, если у нас есть некая функция, принимающая значение типа Int в качестве аргумента, то при её вызове мы будем писать так:

show 6

Однако если эта функция в качестве аргумента принимает объект типа IP_address, то мы должны явно указать это при её вызове:

show (IP_address "127.0.0.1")

Выражение (IP_address “127.0.0.1”) породит объект типа IP_address, “содержащий в себе” строку со значением “127.0.0.1”.

Кстати, метку принято ещё называть конструктором.

Обратите внимание, что имя типа не может начинаться с маленькой буквы. Поэтому такой код:

data ip_address = ip_address String

будет неизбежно отвергнут компилятором.

Класс типов

Проблема в том, что тип IP_address в его нынешнем виде настолько примитивен, что не представляет для нас большого интереса (фактически, никакого интереса не представляет). Ведь мы, создав такой объект, даже не сможем вывести его на консоль. В частности, упомянутая функция show (стандартная функция Haskell, выводящая переданный ей аргумент в строковом виде) с гневом отвергнет аргумент типа IP_address, и правильно сделает: откуда ей знать, как можно представить объект типа IP_address в строковом виде? Она не узнает это до тех пор, пока мы явно ей не расскажем об этом. И вот тут-то на арену и выходят классы типов.

Я понимаю, в выражении “класс типов” слово “класс” невольно режет слух программисту на C++, но потерпите, дальше будет понятнее. Класс типов - это некая обобщённая категория типов, отражающая конкретные общие черты для всех этих типов.

Например, в стандартной библиотеке Haskell есть класс типов Show. Он обобщает все типы, объекты которых могут быть “показаны” (то есть отображены в виде строки). Как именно обобщает, спросите вы? Через общие методы. Вот определение класса Show:

class Show a where
    show :: a -> String

Как видите, тут присутствует один-единственный метод show, принимающий в качестве аргумента объект некоторого типа a и возвращающий некоторое строковое отображение этого объекта.

Но что же такое тип a в этом выражении, спросите вы. Тип a - это любой тип. Рассматривайте его примерно как T в строке template в C++ или как id в Objective-C. Разумеется, в данном случае речь идёт о любом типе, для которого понятие “показать” имеет смысл.

Да, но каким же образом метод show узнает, как ему отобразить объект некоторого конкретного типа в виде строки? Он узнает это через экземпляр класса типов (звучит ещё страшнее, но наберитесь мужества).

Экземпляр класса типов

Если класс типов Show - это абстракция для типов, объекты которых можно “показать”, то экземпляр этого класса представляет собой реальное объяснение о том, как можно “показать” объект того или иного типа. То есть если мы хотим объяснить, как можно “показать” объект типа IP_address, мы должны создать экземпляр класса типов Show для типа IP_address. Сделаем же это:

instance Show IP_address where
    show (IP_address address) = 
        if address == "127.0.0.1"
        then 
            "localhost"
        else 
            address

Как видите, мы использовали ключевое слово instance, а обобщённый тип a заменён здесь на наш конкретный тип IP_address. А дальше следует конкретное определение метода show для нашего типа. Таким образом, класс типов можно рассматривать примерно как чистый интерфейс в C++ (с чисто виртуальными функциями), а экземпляр класса типов - как определение этих чисто виртуальных функций для конкретного типа.

Взглянем ещё раз на определение метода show:

    show (IP_address address) =
        if address == "127.0.0.1"
        then 
            "localhost"
        else 
            address

Первая строка показывает, что метод show принимает объект типа IP_address, который в свою очередь будет “порождён” выражением вида (IP_address address), например, как в примере выше, выражением IP_address “127.0.0.1”. Далее следует простое условие, выводящее слово localhost в случае значения адреса 127.0.0.1.

А теперь мы можем использовать всё это хозяйство следующим образом:

main = 
    putStrLn (show (IP_address "127.0.0.1"))

Теперь-то уж метод show прекрасно знает, как именно “показать” наш объект типа IP_address, поэтому мы получим ожидаемый вывод:

localhost

Если же мы напишем так:

main = 
    -- IP-адрес google.com ;-)
    putStrLn (show (IP_address "46.19.98.158"))

то результат столь же ожидаем:

46.19.98.158

Вот и всё. Теперь вы знаете о пользовательских типах в Haskell. Признаюсь, там не так всё просто, как описано выше, но самый базис вы уже освоили. Продолжение следует…
